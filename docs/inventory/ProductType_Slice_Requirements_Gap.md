# Inventory/ProductType Slice Requirements and Gaps

## Snapshot

- Date: 2026-03-13
- Scope: first non-users slice centered on `Inventory/ProductType`

## What Exists In The Repo Now

| Concern | Repo evidence | Current state |
| --- | --- | --- |
| Canonical endpoint | `_common/include/common/Endpoints.hpp` defines `Endpoints::Inventory::PRODUCT_TYPES = /api/inventory/product-types` | The slice has a stable HTTP path in shared code. |
| Server dispatch | `_server/src/BusinessLogic/BusinessLogic.cpp` registers `inventory` as a real module | Inventory is no longer just a placeholder at module-dispatch level. |
| Business behavior | `_server/src/BusinessLogic/InventoryModule.cpp` handles `GET`, `POST`, `PUT`, and `DELETE` for submodule `product-types` | ProductType CRUD exists on the server side. |
| Validation rules | `InventoryModule` requires `code`, `name`, and `unit`; defaults empty `lastPrice` to `0.0`; normalizes `isImported`; rejects duplicate `code` values | Basic create/update rules are implemented. |
| Persistence | `_server/src/Database/ProductTypeRepository.cpp` and `scripts/create_db.sql` map the slice onto SQLite `ProductType` storage | The slice persists data through the repository layer. |
| Shared entity shape | `_common/AutoGenerated/include/common/Entities/ProductType.hpp` defines `id`, `code`, `barcode`, `name`, `description`, `lastPrice`, `unit`, `isImported` | Shared field names are already aligned across server/common code. |
| Automated checks | `_server/tests/unit/InventoryModuleTest.cpp`, `_server/tests/component/BusinessLogicTest.cpp`, `_server/tests/component/RepositoryIntegrationTest.cpp`, `_server/tests/component/HttpContractTest.cpp`, `_client/tests/unit/ApiManagerTest.cpp`, `_client/tests/unit/ProductTypesModelTest.cpp` | Server, HTTP contract, and client unit-level coverage exist for the slice. |
| Client reachability | `_client/src/Network/ApiManager.*`, `_client/src/Ui/Models/ProductTypesModel.*`, `_client/src/Ui/ViewModels/ProductTypesViewModel.*`, `_client/src/Ui/Views/ProductTypesView.*`, and `_client/src/Ui/MainWindow.cpp` | The slice now reaches the Qt client in the current worktree. |

## Functional Minimum Present

- Open the Product Management tab from the Inventory menu in `MainWindow`.
- Fetch Product Types into the ProductTypes table view.
- Create, edit, and delete Product Types through the current client/server code path.

## Gaps To Reach A User-Usable Slice

| Gap | Evidence | Why it still matters |
| --- | --- | --- |
| No view-level CRUD validation | Current client checks cover `ApiManager`, `ProductTypesModel`, `MainWindow`, and HTTP contracts, but not the `ProductTypesView` create/edit/delete interactions | The user-visible CRUD dialog flow is still only statically reviewed |
| No live fullstack GUI validation | There is no automated test that runs the real Qt client against the real HTTP server | The slice is validated in layers, but not yet in one desktop end-to-end round trip |
| No ProductType search/filter behavior | Product docs promise search/filter, but the current server module only exposes CRUD | The implemented slice is narrower than the target functional spec. |
| No Inventory-record workflow | `InventoryModule` currently serves `product-types` only | This is a ProductType-centered slice, not full inventory management yet. |
